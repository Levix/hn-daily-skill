## 3. this css proves me human

**HN 评论**: [查看讨论](https://news.ycombinator.com/item?id=47281593)

### 中文标题
《这个 CSS 证明我是人类》

### 一句话总结
一种创新的"人类证明"技术，通过检测用户是否使用广告拦截器（如 uBlock Origin）来判断是否为真人，因为机器人通常不会安装这类扩展。

### 详细摘要
在验证码（CAPTCHA）逐渐失效的时代（AI 已经能轻松破解），作者提出了一种新颖的人类验证方法：检测用户是否使用广告拦截器。原理很简单——真正的用户往往会安装 uBlock Origin 等广告拦截扩展，而自动化机器人通常不会。作者通过 CSS 和 JavaScript 巧妙地检测这一特征，实现了无需用户交互的"隐形验证码"。

### 关键要点
- 🚫 **检测逻辑**: 检查 uBlock Origin 等广告拦截器
- 👤 **人类特征**: 真实用户倾向于安装隐私保护扩展
- 🤖 **机器人盲点**: 自动化工具通常不会配置此类扩展
- 🎯 **零摩擦**: 用户无需点击任何验证框
- 🔒 **隐私友好**: 不收集个人数据

### 技术洞察
**实现原理**:
```javascript
// 检测广告拦截器的基本思路
async function detectAdblocker() {
  try {
    // 尝试加载一个已知的广告相关资源
    const response = await fetch('https://pagead2.googlesyndication.com/pagead/js/adsbygoogle.js');
    return false; // 如果能加载，说明没有拦截器
  } catch {
    return true; // 被拦截，说明有拦截器
  }
}
```

**CSS 技巧**:
- 使用 `:has()` 选择器检测特定元素
- 利用广告拦截器的隐藏规则
- 通过计算样式差异判断

**局限性**:
- 假阴性：不使用拦截器的真人
- 假阳性：配置了拦截器的自动化测试环境
- 可被针对性绕过

### 为什么它火了
1. **反直觉**: 用"缺陷"作为"特征"
2. **讽刺意味**: 广告拦截器成了"人类认证"
3. **技术巧妙**: 简单但有效的启发式方法
4. **隐私关注**: HN 社区对 CAPTCHA 替代品感兴趣

### 详细摘要
Helix 是一款新兴的终端文本编辑器，融合了 Vim 的模态编辑效率和现代 IDE 的智能功能。它采用"选择优先"的编辑模型（来自 Kakoune），内置 LSP（语言服务器协议）支持，提供语法高亮、代码补全、诊断信息等功能。作为 Rust 项目，Helix 注重性能和可靠性，同时保持配置简单。

### 关键要点
- ⌨️ **模态编辑**: 类似 Vim 的多模式编辑
- 🌳 **树形选择**: 基于语法树的选择和操作
- 🔧 **内置 LSP**: 无需额外配置即可获得 IDE 功能
- 🦀 **Rust 编写**: 高性能和内存安全
- ⚡ **开箱即用**: 合理的默认配置，无需复杂配置

### 技术洞察
**与 Vim/Neovim 的区别**:
| 特性 | Helix | Vim |
|-----|-------|-----|
| 配置 | 简单 TOML | 复杂 VimScript/Lua |
| LSP | 内置 | 需要插件 |
| 选择模型 | 选择优先 | 操作优先 |
| 插件生态 | 较小但增长中 | 极其丰富 |
| 学习曲线 | 中等 | 陡峭 |

**树形选择 (Tree-sitter)**:
- 基于语法解析的选择
- `ma` - 选择当前函数
- `mi` - 选择函数内部
- `mafa` - 选择外层函数

### 为什么它火了
1. **Vim 疲劳**: 一些用户对 Vim 的复杂配置感到疲惫
2. **Rust 光环**: Rust 项目的质量期望
3. **现代 defaults**: 开箱即用的良好体验
4. **Kakoune 理念**: 选择优先的直观操作

## 6. LLMs work best when the user defines their acceptance criteria first

**原文链接**: [blog.katanaquant.com/p/your-llm-doesnt-write-correct-code](https://blog.katanaquant.com/p/your-llm-doesnt-write-correct-code)  
**HN 评论**: [查看讨论](https://news.ycombinator.com/item?id=47283337)

### 中文标题
《先定义验收标准，LLM 才能写出正确代码》

### 一句话总结
研究表明，当用户先明确代码的测试用例和验收标准时，LLM 生成正确代码的成功率显著提高，这与传统软件开发中"先写测试"的最佳实践一致。

### 详细摘要
作者发现，直接向 LLM 请求代码（如"写一个排序函数"）往往得到次优结果，但当先提供明确的测试用例和验收标准时，LLM 的表现大幅提升。这与测试驱动开发（TDD）的理念不谋而合——明确的期望带来更好的结果。文章提供了实证数据和具体策略，帮助开发者更有效地使用 AI 编程助手。

### 关键要点
- ✅ **先定义标准**: 在请求代码前先定义测试用例
- 📈 **成功率提升**: 有明确验收标准时代码正确率显著提高
- 🧪 **测试驱动**: 与 TDD 方法论一致
- 🎯 **具体优于抽象**: "处理边界情况"不如提供具体边界值
- 📝 **示例驱动**: 提供输入/输出示例比描述更有效

### 技术洞察
**Prompt 工程技巧**:
```
❌ 低效: "写一个验证邮箱的函数"
✅ 高效: "写一个函数验证邮箱，要求：
  1. 必须包含 @ 符号
  2. 域名部分必须包含 .
  3. 不允许连续的点
  4. 测试用例：
     - valid@test.com → true
     - invalid@test → false
     - @test.com → false"
```

**为什么有效**:
1. **减少歧义**: 明确标准消除解释空间
2. **约束搜索空间**: LLM 在更明确的范围内生成
3. **自我验证**: 可以对照标准检查输出
4. **迭代基础**: 失败时有明确的改进方向

### 为什么它火了
1. **实用建议**: 立即可应用的技巧
2. **方法论契合**: 与 TDD 呼应，有理论支撑
3. **AI 编程痛点**: 直击开发者使用 Copilot 等工具的常见问题
4. **可验证**: 提供了数据和实验

## 7. UUID package coming to Go standard library

**原文链接**: [github.com/golang/go/issues/62026](https://github.com/golang/go/issues/62026)  
**HN 评论**: [查看讨论](https://news.ycombinator.com/item?id=47283665)

### 中文标题
《UUID 包即将加入 Go 标准库》

### 一句话总结
Go 团队正在将 UUID（通用唯一识别码）支持加入标准库，这是社区长期以来的需求，将消除对外部依赖的需求。

### 详细摘要
Go 语言标准库即将加入对 UUID（Universally Unique Identifier）的原生支持。UUID 是软件开发中广泛使用的标识符格式，但 Go 开发者一直需要依赖第三方库（如 google/uuid）。这个提案已在 GitHub 上获得广泛关注，预计将在未来的 Go 版本中实现。

### 关键要点
- 📦 **标准库支持**: 不再依赖第三方包
- 🆔 **UUID 版本**: 计划支持 UUID v4、v7 等常用版本
- 🚀 **性能优化**: 标准库实现通常更快
- 🔒 **质量保证**: Go 团队的代码质量保证
- 📋 **API 设计**: 遵循 Go 的简洁哲学

### 技术洞察
**UUID 版本对比**:
| 版本 | 特点 | 适用场景 |
|-----|------|---------|
| v4 | 完全随机 | 通用唯一标识 |
| v7 | 时间排序 + 随机 | 数据库主键（可排序）|
| v6 | 时间排序 | 分布式系统 |

**为什么重要**:
- 几乎每个 Go 项目都需要 UUID
- 消除一个几乎必选的依赖
- 统一社区实现，减少碎片化
- 更好的长期维护保证

### 为什么它火了
1. **普遍需求**: UUID 是几乎所有项目的必需品
2. **去依赖化**: 符合 Go "少即是多"的理念
3. **社区呼声**: 这是长期以来社区投票最高的提案之一
4. **v7 支持**: UUID v7 是较新的可排序 UUID，数据库友好

### 详细摘要
Ki Editor 是一种新型代码编辑器，它打破了传统文本编辑的范式，直接在代码的抽象语法树（AST）层面进行操作。这意味着你不再是在操作字符，而是在操作代码的结构元素——函数、变量、表达式等。这种"结构化编辑"提供了传统编辑器无法实现的精确性和安全性。

### 关键要点
- 🌳 **AST 优先**: 直接操作代码的语法树结构
- ✂️ **智能选择**: 选择"表达式"而非"字符"
- 🔄 **安全重构**: 结构性保证重构的正确性
- 🎯 **语义感知**: 编辑器理解代码含义
- 🔧 **多语言**: 支持多种编程语言

### 技术洞察
**AST 编辑 vs 文本编辑**:
| 操作 | 文本编辑器 | Ki Editor |
|-----|-----------|-----------|

| 提取变量 | 手动编辑 | `extract-variable` |

**优势**:
